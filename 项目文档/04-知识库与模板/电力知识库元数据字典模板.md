# 电力知识库元数据字典模板

## 1. 使用说明

这份文档用于统一电力行业知识库的元数据标准，适用于以下知识对象：

- 专家文献
- 标准规范
- 解决方案
- 产品文档
- 项目案例
- 故障案例
- 工单与运维记录

目标是让知识库具备以下能力：

- 支持精确过滤检索
- 支持多 Agent 路由调用
- 支持按场景、设备、电压等级、地区、项目等维度进行知识复用
- 支持版本治理、审核治理、可信度治理

建议原则：

- 字段名统一
- 字段值标准化
- 枚举优先，自由文本最少化
- 自动抽取与人工审核结合
- 元数据标准独立于具体 RAG 框架

---

## 2. 元数据字段分层

建议把元数据字段拆分为 4 层：

1. 全局通用字段
2. 分类识别字段
3. 行业业务字段
4. 治理运维字段

---

## 9. 元数据管理流程建议

### 9.1 入库流程

1. 上传原始文档
2. 自动识别知识类型
3. 自动抽取基础字段
4. 系统推荐业务标签
5. 人工审核关键字段
6. 校验通过后正式入库
7. 建立版本记录

### 9.2 自动抽取字段建议

适合自动抽取：

- 标题
- 时间
- 标准编号
- 项目名称
- 地区
- 作者/机构

适合模型推荐、人工确认：

- `business_domain`
- `scenario`
- `grid_environment`
- `equipment_type`
- `fault_type`

必须人工确认：

- `reliability_level`
- `review_status`
- `security_level`
- `applicable_boundary`

---

## 10. 校验规则建议

建议至少配置以下校验规则：

- 必填字段不能为空
- 枚举字段必须来自标准字典
- 日期格式必须统一
- 失效日期不能早于生效日期
- 审核状态为 `approved` 的文档必须有责任人和版本号
- 安全级别为 `confidential` 及以上的文档禁止默认开放给通用 Agent
- `fault_case` 类型文档应至少包含 `fault_type` 或 `cause_type`
- `solution` 类型文档应至少包含 `scenario` 和 `applicable_boundary`

---

## 11. 最终建议

建议把元数据标准作为独立资产管理，而不是只写在某个 RAG 框架配置里。

推荐做法：

- 元数据字典保存在独立配置库或主数据表中
- 检索系统只消费标准化后的元数据
- Agent 统一走检索服务，不直接依赖底层文档字段格式

一句话总结：

`知识分类解决“是什么”，元数据标准解决“怎么精确找到、过滤、路由、复用它”。元数据治理本质上是主数据治理，而不是简单的文档标签。`
