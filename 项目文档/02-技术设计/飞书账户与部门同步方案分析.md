# 飞书账户与部门同步方案分析

## 1. 文档目的

本文档用于分析在“售前闭环智能体平台”中接入飞书协同能力时，平台账户体系与飞书账户体系、部门体系之间如何打通，以及员工离职后的账户处置策略。

目标不是一次性完成“完全飞书化登录”，而是为后续：

- 飞书消息/任务流转
- 人员选择与指派
- 部门归属判断
- 权限管理
- 离职处置

建立一套稳定、可演进的身份策略。

## 2. 当前问题

平台当前已经有自建账户、部门、角色、权限体系。

一旦接入飞书，会立刻遇到以下问题：

1. 平台用户与飞书用户无法天然一一对应。
2. 平台里的部门树与飞书部门树可能不一致。
3. 发飞书消息、指派任务时，不知道应该命中哪个飞书用户。
4. 员工调岗、改名、离职后，如果双边各自维护，信息容易漂移。
5. 审计日志、任务责任、部门归属可能出现错位。

因此必须先明确：

**身份主数据由谁负责，权限由谁负责。**

## 3. 核心结论

### 3.1 推荐原则

推荐采用：

**飞书同步身份与部门，平台维护角色与权限。**

即：

- `飞书负责`
  - 用户主档
  - 部门树
  - 用户归属部门
  - 在职/离职状态
  - 基础展示信息

- `平台负责`
  - 角色
  - 权限
  - 模块可见性
  - 数据范围
  - 审计能力
  - 业务动作权限

### 3.2 不推荐方案

不推荐长期维持：

**飞书一套账户、平台一套账户、两边手工维护关联。**

原因：

1. 成本高
2. 易漂移
3. 容易发错人
4. 难以支持后续任务闭环和审计

## 4. 推荐实施路径

### 4.1 阶段一：账户映射

先不改登录方式，先补“飞书身份映射”。

平台用户表建议新增：

- `feishu_user_id`
- `feishu_open_id`
- `feishu_union_id`
- `feishu_primary_department_id`
- `feishu_department_ids`
- `identity_source`
- `feishu_sync_status`
- `feishu_last_sync_at`

这一阶段的目标是：

- 能把平台用户和飞书用户对应起来
- 能把平台里的负责人选择映射到飞书消息接收人
- 为后续部门同步、登录打通做准备

### 4.2 阶段二：飞书用户与部门同步

在平台后端增加同步任务：

- 飞书部门树同步
- 飞书用户同步
- 在职状态同步
- 组织变更同步

此阶段重点：

1. 部门不再主要靠手工维护
2. 用户基础信息不再主要靠平台手工维护
3. 平台仍保留角色、权限配置

### 4.3 阶段三：登录与单点登录评估

在账户映射和部门同步稳定后，再评估是否：

- 允许飞书登录平台
- 将平台登录切换为飞书优先
- 或保留平台登录与飞书映射并存

本阶段不建议在尚未稳定同步前就直接推动。

## 5. 为什么平台权限仍然要独立维护

即使飞书同步了账户和部门，也不建议把复杂业务权限交给飞书。

原因：

1. 平台权限不只是组织关系问题
2. 还包括业务模块权限与操作权限
3. 例如：
   - 谁能查看需求分析
   - 谁能生成方案
   - 谁能查看审计日志
   - 谁能管理知识库
   - 谁能发起飞书流转

因此最稳的边界是：

- `飞书负责“你是谁”`
- `平台负责“你能做什么”`

## 6. 部门同步建议

### 6.1 建议同步的部门信息

- 飞书部门 ID
- 部门名称
- 上级部门
- 部门层级路径
- 部门状态
- 部门负责人（如可获取）

### 6.2 平台侧部门模型建议

平台部门表建议新增或保留：

- `external_source = feishu`
- `external_department_id`
- `external_parent_department_id`
- `sync_status`
- `last_sync_at`

### 6.3 同步策略

建议采用：

- 首次全量同步
- 后续定时增量同步
- 重要变更支持手动重同步

## 7. 用户同步建议

### 7.1 建议同步的用户字段

- 用户名（平台内部可继续保留）
- 姓名
- 展示名
- 手机号
- 邮箱
- 飞书用户 ID
- 主部门
- 所属部门列表
- 在职状态
- 启用/停用状态
- 最近同步时间

### 7.2 平台用户与飞书用户的对应原则

推荐以：

- `feishu_user_id`

作为主映射键。

`手机号/邮箱` 可作为首次匹配辅助字段，但不建议作为长期唯一主键。

## 8. 员工离职后的账户处置建议

这一点必须提前设计，否则后续飞书同步会把离职人员带出一堆权限和消息问题。

### 8.1 原则

员工离职后：

- 平台历史数据必须保留
- 平台登录权限应关闭
- 未完成任务应可转派
- 审计记录应完整保留
- 归档资料不能丢失

### 8.2 推荐处置流程

#### A. 飞书侧状态变化

飞书同步发现用户离职/停用后：

- 平台将其标记为：
  - `employment_status = resigned`
  - `account_status = disabled`

#### B. 平台动作

1. 禁止该用户继续登录
2. 清理有效 token
3. 保留其历史会话、报告、任务发起记录、审计记录
4. 将未完成的跟进任务列入“待转派队列”
5. 必要时通知部门管理员或平台管理员处理

#### C. 数据归属

离职人员历史创建的：

- 客户沟通会话
- 需求分析报告
- 解决方案结果
- 审计日志

原则上不删除，只改变账号状态和任务责任归属。

### 8.3 不建议的做法

不建议：

- 离职后直接硬删除平台账户
- 删除历史数据
- 让离职员工仍保留飞书映射和平台有效权限

## 9. 飞书同步后的平台账户形态建议

### 9.1 推荐最终形态

平台账户建议变成：

**“飞书身份 + 平台授权”**

这意味着：

- 账户主档和部门来自飞书
- 平台自己控制角色和权限
- 平台可继续补充业务字段

例如：

- 是否为平台管理员
- 是否可访问知识库管理
- 是否可发起方案生成
- 是否可发起飞书流转
- 是否可查看审计日志

### 9.2 平台仍可补充的业务字段

- 岗位
- 业务角色
- 数据范围
- 权限标签
- 最近一次飞书同步时间
- 组织映射异常标记

## 10. 对当前平台的建议改造点

### 10.1 数据模型

建议新增或补充：

- User 增加飞书映射字段
- Department 增加飞书映射字段
- 同步日志表
- 组织同步任务表（可选）

### 10.2 后台管理能力

建议新增：

- 飞书用户映射查看
- 飞书部门映射查看
- 手动重同步
- 映射冲突提示
- 离职用户待转派任务视图

### 10.3 权限管理界面

建议明确区分：

- 身份来源：飞书同步
- 权限来源：平台配置

避免管理员误以为飞书部门会自动决定全部权限。

## 11. MVP 阶段建议

### 11.1 本期建议做

1. 飞书用户映射字段
2. 飞书部门映射字段
3. 基础用户同步
4. 基础部门同步
5. 离职用户自动停用
6. 未完成任务待转派机制

### 11.2 本期不建议做

1. 复杂双向组织同步
2. 平台权限由飞书自动推导
3. 完整单点登录替换
4. 离职后自动全量任务转派

## 12. 一句话总结

如果飞书要成为售前闭环协同层，推荐路线不是“飞书和平台各管各的账户”，而是：

**飞书同步身份与部门，平台继续维护角色与权限，并为离职员工设计停用、保留、转派的完整处置机制。**
