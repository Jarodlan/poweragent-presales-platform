# 客户需求分析智能体 数据库表结构设计草案

## 1. 文档目的

本文档用于定义“客户需求分析智能体”在 Django 平台层中的核心数据表结构，为后端建模、接口开发、权限控制、审计追踪与后续扩展提供统一基础。

本设计遵循以下原则：

1. 优先复用现有用户、审计、任务体系
2. 支持实时转写与分段保存
3. 支持阶段性整理与最终需求分析报告
4. 支持知识库开关与跨行业场景
5. 支持后续与解决方案生成智能体联动

---

## 2. 表结构总览

建议新增一个业务域 app，例如：

- `customer_demand`

建议至少包含以下表：

1. `customer_demand_session`
2. `customer_demand_participant`
3. `customer_demand_segment`
4. `customer_demand_stage_summary`
5. `customer_demand_report`
6. `customer_demand_analysis_task`
7. `customer_demand_attachment`（可选）

其中：

- `session`：客户沟通主会话
- `segment`：ASR 转写分段
- `stage_summary`：阶段性整理结果
- `report`：最终需求分析结果
- `analysis_task`：分析任务与执行状态

---

## 3. customer_demand_session

### 3.1 表用途

用于保存一次客户需求沟通的主记录，是所有分段、分析、导出、审计的主实体。

### 3.2 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | UUID / bigint | 主键 |
| `owner_id` | FK -> `accounts_user` | 会话归属用户 |
| `department_id` | FK -> `accounts_department` nullable | 归属部门 |
| `customer_name` | varchar(200) | 客户名称 |
| `session_title` | varchar(255) | 会话标题 |
| `industry` | varchar(100) | 行业标签 |
| `region` | varchar(100) | 地区 |
| `topic` | varchar(255) | 本次沟通主题 |
| `customer_type` | varchar(50) nullable | 客户类型 |
| `knowledge_enabled` | boolean | 是否启用知识库 |
| `knowledge_scope` | jsonb / json nullable | 指定知识库范围或配置 |
| `status` | varchar(32) | 会话状态 |
| `recording_started_at` | datetime nullable | 开始记录时间 |
| `recording_stopped_at` | datetime nullable | 结束记录时间 |
| `analysis_started_at` | datetime nullable | 开始分析时间 |
| `analysis_finished_at` | datetime nullable | 分析完成时间 |
| `raw_segment_count` | int | 原始分段数 |
| `normalized_segment_count` | int | 校验后分段数 |
| `latest_stage_version` | int | 当前阶段整理版本号 |
| `latest_report_id` | FK -> `customer_demand_report` nullable | 当前有效报告 |
| `remarks` | text nullable | 备注 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 3.3 状态枚举建议

- `draft`
- `recording`
- `paused`
- `closed`
- `analyzing`
- `completed`
- `failed`
- `archived`

### 3.4 索引建议

1. `owner_id + created_at`
2. `department_id + created_at`
3. `status + updated_at`
4. `customer_name`
5. `industry`
6. `region`

| `job_title` | varchar(100) nullable | 职位 |

| `is_internal` | boolean | 是否内部人员 |

### 4.3 participant_type 枚举建议

- `sales`
- `presales`

- `project_manager`

## 5. customer_demand_segment

### 5.1 表用途

保存实时 ASR 的分段结果，是本系统最关键的数据明细表。

### 5.2 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | UUID / bigint | 主键 |
| `session_id` | FK -> `customer_demand_session` | 所属会话 |
| `sequence_no` | int | 分段序号 |
| `speaker_label` | varchar(32) nullable | 说话人标签 |
| `raw_text` | text | 原始 ASR 文本 |
| `normalized_text` | text nullable | 语义校验后文本 |
| `final_text` | text nullable | 最终展示文本，可与 normalized 一致 |
| `asr_provider` | varchar(50) nullable | ASR 提供方 |
| `llm_provider` | varchar(50) nullable | 语义校验模型提供方 |
| `confidence_score` | numeric(5,4) nullable | ASR 或综合置信度 |
| `semantic_score` | numeric(5,4) nullable | 语义一致性评分 |
| `review_flag` | boolean | 是否需要人工确认 |
| `issues_json` | jsonb / json nullable | 校验问题列表 |
| `raw_start_ms` | bigint nullable | 原始音频片段起始毫秒 |
| `raw_end_ms` | bigint nullable | 原始音频片段结束毫秒 |
| `segment_status` | varchar(32) | 分段状态 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 5.3 segment_status 枚举建议

- `draft`
- `asr_ready`
- `normalized`
- `review_required`
- `discarded`

### 5.4 索引建议

1. `session_id + sequence_no`
2. `session_id + created_at`
3. `review_flag`

### 8.2 字段设计

| 字段名 | 类型 | 说明 |
|---|---|---|
| `id` | UUID / bigint | 主键 |
| `session_id` | FK -> `customer_demand_session` | 所属会话 |
| `task_type` | varchar(32) | 任务类型 |
| `status` | varchar(32) | 任务状态 |
| `current_step` | varchar(100) nullable | 当前步骤 |
| `current_step_label` | varchar(100) nullable | 当前步骤中文名 |
| `progress` | int nullable | 进度百分比 |
| `request_payload` | jsonb / json nullable | 请求参数 |
| `result_payload` | jsonb / json nullable | 结果摘要 |
| `error_message` | text nullable | 错误信息 |
| `started_by_id` | FK -> `accounts_user` nullable | 发起人 |
| `started_at` | datetime nullable | 开始时间 |
| `finished_at` | datetime nullable | 结束时间 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

### 8.3 task_type 枚举建议

- `stage_summary`
- `final_analysis`

### 8.4 status 枚举建议

## 10. 与现有表的关系

### 10.1 accounts_user

通过 `owner_id`、`created_by_id`、`started_by_id` 等字段关联。

### 10.2 accounts_department

会话可挂靠所属部门，用于后续统计与权限扩展。

### 10.3 audit_auditlog

建议所有关键操作写入审计日志：

1. 创建会话
2. 开始记录
3. 结束记录
4. 触发整理
5. 启动分析
6. 导出报告
7. 删除/归档会话

## 15. 结论

客户需求分析智能体的数据库设计应围绕：

1. 会话主记录
2. 实时分段文本
3. 阶段整理
4. 最终分析报告
5. 任务状态追踪

这五个核心对象展开。

推荐先按最小闭环建表，保证：

- 记录
- 整理
- 分析
- 回看
- 权限

五条链同时成立，再逐步补附件、参与人、导出、联动等增强能力。
