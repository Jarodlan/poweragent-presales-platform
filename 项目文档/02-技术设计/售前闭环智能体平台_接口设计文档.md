# 售前闭环智能体平台 接口设计文档

## 1. 文档目标

本文档用于定义“售前闭环智能体平台”一期 MVP 的后端接口，支撑以下能力：

1. 从客户需求分析报告发起跟进任务
2. 从解决方案结果发起跟进任务
3. 查询与管理售前跟进任务
4. 查看归档记录
5. 手动触发飞书发送
6. 查看飞书发送记录
7. 预留飞书同步任务入口
8. 支持飞书个人任务授权与卡片交互

本文档遵循以下原则：

- 以平台内任务主数据为核心
- 飞书只作为协同发送层
- 所有发送动作需要留痕
- 重要流转支持人工确认后触发

## 4. 任务接口设计

## 4.1 从需求分析报告发起跟进任务

### `POST /api/v1/presales/tasks/from-demand-report`

### 作用

从客户需求分析报告创建一条建议任务，供人工确认或直接进入后续跟进。

### 请求体示例

```json
{
  "report_id": "a5d5d7d8-1111-2222-3333-444444444444",
  "task_title": "跟进宁波港口工业园储能运营需求",
  "task_description": "根据需求分析报告，安排下一轮客户澄清与方案准备。",
  "customer_name": "宁波港口工业园",
  "assignee_user_id": 12,
  "priority": "high",
  "due_at": "2026-03-26T18:00:00+08:00",
  "next_follow_up_at": "2026-03-25T10:00:00+08:00",
  "collaborator_user_ids": [15, 18],
  "payload_json": {
    "need_feishu_delivery": true
  }
}
```

### 返回示例

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "task": {
      "id": "0a0a0a0a-1111-2222-3333-444444444444",
      "task_title": "跟进宁波港口工业园储能运营需求",
      "source_type": "customer_demand_report",
      "status": "pending"
    }
  }
}
```

## 4.2 从解决方案结果发起跟进任务

### 作用

从解决方案结果创建跟进任务，用于内部评审、客户汇报、二次补充材料等。

```json
{
  "source_id": "62941e8a-908e-4d6b-b7a7-c4b392b07db6",
  "source_title": "宁波分布式储能聚合运营智能体解决方案",
  "task_title": "安排储能聚合方案内部评审",
  "task_description": "请售前与技术支持一起评审方案内容并准备客户汇报材料。",
  "customer_name": "宁波港口工业园",
  "assignee_user_id": 18,
  "priority": "medium",
  "due_at": "2026-03-27T14:00:00+08:00",
  "collaborator_user_ids": [12],
  "payload_json": {
    "solution_scene": "storage_aggregation_solution"
  }
}
```

## 4.8 手动发送到飞书

### `POST /api/v1/presales/tasks/{task_id}/send-feishu`

### 请求体示例

```json
{
  "target_type": "user",
  "target_id": "ou_xxxxx",
  "message_type": "interactive_card",
  "message_payload": {
    "text": "请优先跟进客户第二轮需求澄清。"
  }
}
```

补充说明：

- 前端当前实现为多成员 + 多群聊并行发送
- 成员通过部门树勾选
- 同一飞书账号若属于多个部门，会在多个部门下分别出现
- 后端按每个目标分别创建发送记录

### 返回示例

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "delivery": {
      "id": "1b1b1b1b-1111-2222-3333-444444444444",
      "delivery_status": "queued"
    }
  }
}
```

### `GET /api/v1/presales/feishu/recipients`

返回：

- `departments`
- `users`
- `groups`

说明：

- `users` 以“部门 + 飞书账号”的组合返回
- 用于支持同一飞书账号在不同部门下分别勾选

## 6.4 飞书个人任务授权接口

### `GET /api/v1/presales/feishu/user-auth/status`

返回：

- `authorized`
- `auth_status`
- `authorized_at`

### `GET /api/v1/presales/feishu/user-auth/start`

返回：

- `authorize_url`

### `GET /api/v1/presales/feishu/user-auth/callback`

用途：

- 处理飞书用户 OAuth 回调
- 保存用户 access token / refresh token

## 6.5 飞书卡片回调接口

### `POST /api/v1/presales/feishu/card-callback`

当前支持动作：

- `create_personal_feishu_task`

处理策略：

1. 回调先快速响应飞书
2. 后台异步创建个人飞书任务
3. 成功后回写绿色成功态卡片
4. 重复点击命中幂等时直接返回成功态

## 9. 错误处理建议

建议统一错误策略：

1. 平台任务先落库，再执行飞书发送
2. 飞书发送失败不回滚任务创建
3. 失败结果写入 `feishu_delivery_record`
4. 前端显示“任务已创建，但飞书发送失败”
5. 若接收人未完成飞书个人授权，卡片转个人任务时应给出明确提示

## 11. 总结

一期接口设计建议围绕“任务主数据先落地、飞书协同后接入”的思路推进：

1. 先把售前任务流转跑通
2. 再把飞书发送和同步能力接上
3. 所有关键动作都保留平台主记录与审计痕迹

这样既能快速验证售前闭环的业务价值，又不会因为外部系统耦合过深而拖慢交付节奏。
