# 售前闭环智能体平台 数据库表结构设计草案

## 1. 文档目标

本文档用于定义“售前闭环智能体平台”在 Django 平台层中的核心数据表结构，为后续以下能力提供统一数据基础：

1. 需求分析结果流转到飞书
2. 解决方案结果流转到飞书
3. 自动创建售前跟进任务
4. 回访提醒与任务协同
5. 资料归档与检索
6. 飞书用户与部门同步
7. 员工离职后的账户停用与待转派处理

本设计遵循以下原则：

1. 优先复用现有账户、权限、审计与业务对象
2. 飞书承担身份与部门主数据的同步来源
3. 平台继续维护角色、权限、任务与归档主数据
4. 所有外部发送与同步动作必须可追踪、可审计、可重试
5. 本期先支持售前闭环 MVP，不扩展为完整 CRM

---

## 2. 表结构总览

建议新增一个业务域 app，例如：

- `presales_center`

建议至少包含以下表：

1. `presales_task`
2. `presales_task_activity`
3. `presales_archive_record`
4. `presales_followup_reminder`
5. `feishu_delivery_record`
6. `feishu_sync_job`
7. `feishu_department_sync_snapshot`
8. `feishu_user_sync_snapshot`

同时建议在现有账户体系中补充飞书映射字段：

1. `accounts_user`
2. `accounts_department`

其中：

- `presales_task`：售前跟进任务主表
- `presales_task_activity`：任务流转轨迹与操作审计
- `presales_archive_record`：售前资料归档索引
- `presales_followup_reminder`：回访提醒与到期提醒
- `feishu_delivery_record`：消息发送记录
- `feishu_sync_job`：飞书同步任务记录
- `snapshot` 表：记录飞书同步结果快照，便于差异分析与追溯

- `presales_task.source_type = solution_result`

## 14. 员工离职后的数据处置建议

### 14.1 不删除的内容

员工离职后，以下内容保留：
- 历史任务
- 历史发送记录
- 历史归档记录
- 历史审计记录

### 14.2 需要变更的内容

建议自动执行：
1. `accounts_user.is_active = false`
2. `employment_status = resigned`
3. 清理 Token 与登录态
4. 将 `status in (pending, in_progress)` 的任务加入待转派清单
5. 记录一条离职相关的同步任务和审计记录

### 14.3 待转派机制建议

可选两种实现：

#### A. 直接回退到部门管理员
- `assignee_user_id = 部门管理员`
- `status = reassigned`

#### B. 进入待转派池
- 新增 `pending_reassignment = true`
- 由平台管理员/部门管理员手动处理

MVP 更推荐 B，风险更低。

## 16. 总结

售前闭环智能体平台的数据层设计，应坚持以下原则：

1. `飞书同步身份与部门`
2. `平台维护角色与权限`
3. `平台维护任务、归档与发送记录主数据`
4. `所有飞书交互都必须可追踪、可审计、可失败回放`
5. `离职用户只停用，不删除历史业务数据`

这样既能尽快打通“需求分析 -> 方案 -> 飞书协同 -> 跟进任务 -> 归档”的内部闭环，又为后续 CRM 和客户画像扩展保留了清晰的数据基础。
