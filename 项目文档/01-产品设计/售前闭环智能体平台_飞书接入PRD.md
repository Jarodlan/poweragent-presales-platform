# 售前闭环智能体平台_飞书接入PRD

## 1. 文档目的

本文档用于明确当前项目在既有能力基础上，向“售前闭环智能体平台”演进时，优先接入飞书的产品目标、范围边界、核心流程、角色价值与 MVP 实施方案。

当前平台已具备两项核心能力：

1. `客户需求分析智能体`
2. `电力行业解决方案生成智能体`

当前这一方向已进入“售前闭环中心 + 飞书协同”的实现阶段，已落地的一期能力包括：

- `售前闭环中心` 模块入口与任务控制台
- 从 `需求分析报告` / `解决方案结果` 一键创建售前任务
- 售前任务发送飞书 `交互卡片`
- 成员树按部门展开、多成员与群聊同时勾选发送
- 飞书用户/部门同步、平台账号合并、部门合并
- 飞书个人任务授权
- 飞书卡片一键转为接收人的个人飞书任务

下一步目标不是继续孤立增强单个智能体，而是打通从客户沟通到内部协同执行的闭环，包括：

- 会中记录与需求分析
- 解决方案生成
- 跟进任务流转
- 会后提醒与回访安排
- 附件与资料沉淀
- 后续 CRM / 客户画像能力预留

## 2. 产品背景

当前售前工作存在几个典型断点：

1. 客户现场沟通结束后，需求整理、行动分派、方案编制往往分散在多个工具里完成。
2. 需求分析报告和解决方案虽然已能生成，但后续跟进动作仍依赖人工复制、通知和记忆。
3. 会议纪要、录音、附件、方案和客户信息没有稳定沉淀为统一资产。
4. 下次回访提醒、内部协同分派、关键资料归档容易遗漏，影响售前推进效率。

飞书作为内部协同载体，适合作为当前阶段的优先接入对象，用于完成：

- 内部通知
- 任务提醒
- 协同流转
- 文档与附件沉淀
- 表格化跟进记录

## 3. 产品定位

本次扩展后的产品定位为：

**面向售前、售前支持与技术支持团队的“售前闭环智能体平台”。**

平台中的两个业务智能体继续保留：

1. `客户需求分析智能体`
2. `解决方案生成智能体`

飞书接入后，平台新增一层：

**售前协同执行层**

它不替代智能体，而是承接智能体输出后的组织动作。

## 4. 目标用户

### 4.1 直接使用者

- 销售人员
- 售前工程师
- 技术支持人员
- 部门管理者

### 4.2 间接受益者

- 产品经理
- 方案经理
- 管理层

## 5. 目标价值

### 5.1 对业务的价值

1. 缩短从客户沟通到内部跟进执行的时间。
2. 减少需求理解、任务分配和资料沉淀过程中的信息损耗。
3. 提升售前过程的可追踪性和协作透明度。
4. 为后续 CRM 和客户画像建设打下统一的数据入口。

### 5.2 对组织的价值

1. 将客户沟通内容从“个人记忆”转化为“组织资产”。
2. 将智能体输出从“参考内容”推进为“可执行任务”。
3. 为管理层提供对售前推进过程的可见性。

## 6. 为什么当前阶段优先接入飞书

本阶段优先接入飞书，而不是优先做 CRM，有三个原因：

1. 当前最迫切的问题是内部闭环，而不是完整客户经营系统。
2. 飞书更适合承接消息通知、协同提醒、文档链接、表格沉淀等场景。
3. 飞书接入成本更低，能尽快验证从“智能体输出”到“协同执行”的价值。

当前判断：

- `飞书`：优先用于内部协同闭环
- `CRM`：作为后续主数据沉淀层逐步建设
- `客户画像`：作为 CRM / 客户资产体系的中期能力，不纳入本次 MVP 强承诺

## 6.1 账户与部门同步原则

飞书接入不是单纯增加消息发送能力，还会带来身份和组织体系问题。

本项目建议采用：

**飞书同步身份与部门，平台维护角色与权限。**

即：

- `飞书负责`
  - 用户主档
  - 部门树
  - 用户归属部门
  - 在职/离职状态

- `平台负责`
  - 角色
  - 权限
  - 模块可见性
  - 数据范围
  - 审计与业务授权

本期不强制承诺飞书单点登录替换平台登录，但需要为后续打通预留：

- 飞书用户映射字段
- 飞书部门映射字段
- 基础同步任务

## 6.2 员工离职后的账户处置原则

员工离职后应满足以下原则：

1. 平台登录权限关闭
2. 有效 token 清理
3. 历史需求分析、方案、会话、审计日志保留
4. 未完成的跟进任务可进入待转派队列
5. 历史资料和归档记录不删除

因此，飞书同步不仅用于“找得到人”，也用于：

- 保证人员状态一致
- 降低账号漂移
- 避免消息误发
- 支撑任务责任迁移

## 7. 本期建设范围

### 7.1 纳入范围

#### A. 智能体结果流转到飞书

1. 需求分析报告生成后，可一键发起飞书内部跟进
2. 解决方案生成完成后，可一键发起飞书内部跟进
3. 支持先人工确认和编辑，再发起飞书流转

#### B. 自动创建售前跟进任务

系统根据需求分析结果或方案结果，自动生成建议跟进动作，例如：

- 补充客户侧数据清单
- 安排第二轮需求沟通
- 安排现场勘查
- 指定售前方案编写负责人
- 安排技术验证

#### C. 飞书通知与提醒

支持向指定人员或群组发送：

- 新的客户需求分析结果通知
- 新生成解决方案通知
- 新跟进任务提醒
- 下次回访提醒

本期实际落地形态为：

- 飞书 `交互卡片消息`
- 支持个人和群聊同时触达
- 接收人在飞书中点击后，可将售前任务转为自己的飞书任务

#### D. 附件与资料归档

支持将与会话相关的材料统一沉淀，包括：

- 沟通录音
- 沟通分段文本
- 阶段整理
- 正式需求分析报告
- 解决方案 Markdown / PDF / HTML
- 上传的补充附件

#### E. 任务状态回写

平台内需要保留跟进任务状态，用于后续：

- 查看是否已发送到飞书
- 查看是否已确认/已完成
- 查看是否已提醒

### 7.2 不纳入本期范围

1. 完整 CRM 自建
2. 客户联系人体系完整建模
3. 商机漏斗、合同、报价、回款等 CRM 深水区功能
4. 企业微信 / 钉钉并行接入
5. 飞书深度单点登录改造
6. 自动无确认写入外部系统

## 8. 核心产品原则

### 8.1 人工确认优先

所有对外部协同系统的写入动作，默认采用：

`智能体生成建议 -> 人工确认/编辑 -> 再流转`

不做无确认自动发送。

### 8.2 飞书承担协同，不承担主业务逻辑

飞书是协同执行层，不是主业务数据库。

平台内部仍应保留：

- 会话
- 报告
- 附件
- 任务状态
- 审计日志

### 8.3 先做内部闭环，再考虑 CRM 外延

优先打通：

`沟通 -> 分析 -> 方案 -> 任务 -> 提醒 -> 归档`

客户画像和 CRM 主档放到后续阶段。

## 9. 关键业务流程

### 9.1 流程一：需求分析结果流转到飞书

1. 用户在客户需求分析智能体中完成一轮沟通记录
2. 生成阶段整理与正式需求分析报告
3. 点击 `发起内部跟进`
4. 系统生成建议任务清单
5. 用户确认：
   - 跟进事项
   - 负责人
   - 截止时间
   - 是否发送到飞书群/个人
6. 系统同步飞书消息或任务
7. 平台内部保存流转记录

#### 9.1.1 接收人在飞书内的后续动作

1. 接收人在飞书中收到售前任务卡片
2. 点击 `转为我的飞书任务`
3. 若未完成飞书个人授权，平台先引导其完成授权
4. 授权完成后，平台以该接收人的飞书身份创建飞书个人任务
5. 卡片更新为绿色成功态
6. 同一接收人再次点击时不重复创建

### 9.2 流程二：解决方案结果流转到飞书

1. 用户在解决方案智能体中生成方案
2. 点击 `发起方案跟进`
3. 系统自动带入：
   - 客户名称
   - 会话标题
   - 需求主题
   - 方案摘要
4. 用户确认后，发送到飞书
5. 飞书中生成通知/待办/表格记录

### 9.3 流程三：回访提醒

1. 用户在任务流转时填写预计回访时间
2. 系统保存回访提醒任务
3. 到期前自动发送飞书提醒
4. 用户可以在平台标记：
   - 已回访
   - 延期
   - 关闭

## 10. 功能设计

### 10.1 新增模块：售前协同中心

建议在统一模块入口页中新增：

- `售前协同中心`

其功能包括：

- 跟进任务列表
- 回访提醒
- 飞书发送记录
- 附件归档查看
- 飞书个人任务授权状态
- 飞书同步任务查看与触发

### 10.2 智能体结果页新增动作

#### 在需求分析报告页新增

- `发起内部跟进`
- `发送到飞书`
- `归档资料`

#### 在解决方案结果页新增

- `发起方案跟进`
- `发送到飞书`
- `归档到客户资料`

### 10.3 飞书流转确认弹窗

弹窗建议包含：

- 标题
- 流转摘要
- 建议任务列表
- 负责人选择
- 截止时间
- 回访时间
- 是否附带报告/方案链接
- 是否附带附件清单
- 发送范围（个人 / 群）

当前实现细化为：

- 成员树状选择
  - 按部门展开
  - 勾选部门下成员
- 群聊多选
- 成员和群聊可在同一次发送中同时勾选
- 若同一飞书账号属于多个部门，可在多个部门下分别展示与勾选

### 10.4 附件归档卡片

每个客户沟通会话和方案会话都要支持：

- 查看已有附件
- 查看录音源文件
- 下载原文件
- 标记是否已归档到飞书文档/云端目录

## 13. MVP 验收标准

### 13.1 必须实现

1. 需求分析报告可以发起内部跟进
2. 解决方案结果可以发起内部跟进
3. 支持人工确认编辑后发送到飞书
4. 支持保存跟进任务和提醒时间
5. 支持会后查看飞书发送记录
6. 支持录音、报告、方案等资料的本地归档状态展示
7. 支持飞书用户/部门基础映射能力预留
8. 支持离职用户停用与待转派处理机制预留
9. 支持接收人在飞书卡片中将任务转为自己的飞书任务
10. 飞书任务中的截止时间应与平台到期时间一致

### 13.2 可选但推荐

1. 飞书群通知与个人通知双模式
2. 附件/报告链接自动拼装
3. 回访提醒自动发送

## 14. 实施阶段建议

### 阶段 1：飞书消息与任务闭环

目标：

- 需求分析报告 -> 飞书提醒
- 方案生成 -> 飞书提醒
- 建立任务记录与回访提醒

### 阶段 2：资料归档与售前协同中心

目标：

- 平台内查看归档状态
- 统一附件归档视图
- 跟进任务状态管理

### 阶段 3：轻量 CRM 主档与客户画像预留

目标：

- 客户基础档案页
- 客户沟通与方案历史聚合
- 客户画像字段预留

## 16. 一句话总结

本次飞书接入的核心目标，不是做一个“消息推送插件”，而是把现有智能体平台进一步打通为：

**客户沟通 -> 需求分析 -> 解决方案 -> 跟进任务 -> 回访提醒 -> 资料归档**

的售前闭环平台。
