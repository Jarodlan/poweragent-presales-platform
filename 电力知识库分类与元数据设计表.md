# 电力知识库分类与元数据设计表

## 1. 设计目的

这份表用于指导电力知识库在实际落地时，如何做到：

- 分类清晰
- 字段统一
- 入库规范
- 便于 Agent 检索路由
- 便于后续扩展到多场景、多项目、多团队协作

---

## 2. 一级分类设计

| 一级分类 | 编码 | 说明 | 典型内容 | 是否建议独立知识库 |

| 标准规范 | `standard` | 法规、规程、技术标准 | 国标、行标、企标、调度规程 | 是 |

## 4. 通用字段设计表

| 字段名 | 中文名 | 是否必填 | 数据类型 | 示例 | 检索用途 |
|---|---|---|---|---|---|

## 9. 检索与 Agent 使用建议

### 9.1 检索过滤建议

故障诊断 Agent 常用过滤：

- `knowledge_type in [fault_case, case_report, standard]`
- `business_domain = distribution_network`
- `scenario includes fault_diagnosis`
- `voltage_level includes 10kV`
- `equipment_type includes switch`

规划设计 Agent 常用过滤：

- `knowledge_type in [solution, case_report, standard, paper]`
- `business_domain in [planning, distribution_network]`
- `scenario includes grid_planning`

功率预测 Agent 常用过滤：

- `knowledge_type in [paper, solution, case_report, operation_record]`
- `scenario includes renewable_power_forecasting`
- `renewable_type includes pv or wind`

### 9.2 回答时的可信度约束建议

- 默认优先引用 `approved` + `authoritative/project_verified` 文档
- `reference_only` 文档可用于辅助，不建议作为唯一结论来源
- 高敏案例默认不开放给通用问答 Agent

---

## 10. 最终落地建议

建议实际项目中同时维护三份资产：

1. 分类体系表
2. 元数据字段字典
3. 标签枚举值标准表

如果这三份资产清楚，知识库框架可以替换，但知识治理体系不会乱。

一句话总结：

`知识库分类决定知识放在哪，元数据决定知识能不能被精确找到、被正确路由、被安全复用。`
